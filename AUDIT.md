## 🔴 КРИТИЧНЫЕ проблемы (надо чинить в первую очередь)

### C1. Имя секрета `GITHUB_TOKEN` — это мина
**Файлы:** [scripts/generate_post.py:22](scripts/generate_post.py#L22), [.github/workflows/generate.yml:37](.github/workflows/generate.yml#L37)

```python
GITHUB_TOKEN = os.environ["GITHUB_TOKEN"]
```

`GITHUB_TOKEN` — **зарезервированное** имя в GitHub Actions: Actions автоматически создаёт этот токен с правами на репозиторий. В workflow вы перезаписываете его секретом `GH_MODELS_TOKEN`, но:
1. CLAUDE.md явно запрещает использовать это имя.
2. Любой шаг workflow до `env:` блока (например, `checkout`) увидит **настоящий** `GITHUB_TOKEN` Actions, а Python-скрипт — подменённый. Это путаница.
3. При локальном запуске разработчик не поймёт, что класть в эту переменную.

**Решение:** переименовать в `GH_MODELS_TOKEN` везде (python + workflow). Это базовый рефакторинг, делается за минуту.

---

### C2. Race condition между `generate` и `check_approvals`
**Файлы:** оба workflow ([generate.yml:52-57](.github/workflows/generate.yml#L52-L57), [check_approvals.yml:44-48](.github/workflows/check_approvals.yml#L44-L48))

Оба workflow:
1. Читают `data/*.json`
2. Делают `git stash` → `git pull --rebase` → `git stash pop` → `commit` → `push`

Если они запустятся одновременно (например, генерация в 13:00, а check каждые 10 минут), один из `git push` упадёт из-за non-fast-forward, а `git stash pop` может оставить файлы в смешанном состоянии. `concurrency.group` стоит **разный** для каждого workflow — они не блокируют друг друга.

**Решение:** объединить `concurrency.group` (один общий `data-write`), либо вынести данные в БД (это T1.2 — даёт основное лечение).

---

### C3. Голые `except Exception` глотают ошибки
**Файлы:** [generate_post.py:83-84](scripts/generate_post.py#L83-L84), [check_approvals.py:41-42](scripts/check_approvals.py#L41-L42)

```python
def notify_moderator(text):
    try:
        requests.post(...)
    except Exception:
        pass
```

Любая ошибка (включая `KeyboardInterrupt` в старых питонах, JSON-decode в ответе, network timeout) проглатывается без следа. Если Telegram API лежит — пользователь не узнает.

**Решение:** ловить только `requests.RequestException`, логировать (в Sentry — T1.3), не глотать `KeyboardInterrupt`/`SystemExit`.

---

### C4. `posted_ids` обрезается до 500 — баг с дубликатами
**Файл:** [generate_post.py:540, 565](scripts/generate_post.py#L540)

```python
posted_ids = posted_ids[-500:]
```

При 7 RSS-источниках × 5 статей × 3 запуска в день = до 105 статей/день только просканированных. На 500-м элементе старые ID начинают вылетать → бот может повторно опубликовать ту же новость через 5 дней.

**Решение:** хранить ID минимум 30 дней (по дате, не по количеству). После миграции на БД (T1.2) это решается естественно.

---

### C5. Картинка в `pending.json` хранится URL-ом, а не файлом
**Файл:** [generate_post.py:448-453](scripts/generate_post.py#L448-L453)

```python
return f"https://image.pollinations.ai/prompt/{encoded}?seed={seed}..."
```

`pollinations.ai` генерирует картинку **на лету** по URL. Telegram при `sendPhoto` грузит её по URL. Если:
- между генерацией поста и одобрением Pollinations недоступен → пост опубликуется без картинки или вообще упадёт;
- сервис изменит ответ для того же seed → итоговая публикация будет с другой картинкой, чем модератор одобрял.

**Решение:** скачивать картинку при генерации, сохранять file_id Telegram (через `sendPhoto` в скрытый чат / в момент отправки модератору) и использовать его при публикации.

### C6. Нет проверки наличия обязательных env-переменных
**Файл:** [generate_post.py:19-22](scripts/generate_post.py#L19-L22)

```python
BOT_TOKEN = os.environ["TELEGRAM_BOT_TOKEN"]

Падение `KeyError` на старте — ок, но не диагностично. Не понятно, какие переменные нужны для нового канала, какие опциональны.

## 🟡 ВАЖНЫЕ проблемы

### W1. Дубль кода между двумя скриптами
`load_json`, `save_json`, `notify_moderator`, `tg`, `DATA_DIR`, `PENDING_FILE` — есть в обоих файлах. При изменении формата хранения придётся править в двух местах.

**Решение:** общий модуль `scripts/common.py` или пакет `bot/`.

---

### W2. Промпты захардкожены в .py-файле
Любое изменение промпта = коммит в Python. Нет версионирования, A/B-тестов невозможны.

**Решение:** T2.5 (промпты в БД с версиями). Промежуточно — вынести в `prompts/*.md`.

---

### W3. `call_ai` — нет ограничения общего таймбюджета
Цикл `for model in MODELS: for attempt in range(3): time.sleep(20 * (attempt + 1))`. На rate-limit это даёт до 2 × 3 × 60 = 360 секунд блокировки + сами запросы. GitHub Actions job-таймаут 360 минут, но всё равно: при массовом rate-limit бот зависнет.

**Решение:** общий бюджет (deadline) на весь процесс генерации — например, 5 минут. T1.4 (tenacity с `stop_after_delay`).

---

### W4. `detect_rubric` на ключевых словах — слабо
Срабатывает на английские слова в русских статьях («launch», «tool»), и не различает, что «уволил» в контексте «не уволил» — это другое. На многоязычных каналах ломается полностью.

**Решение:** определять рубрику через LLM (один промпт), либо отдельный classifier — T2.7.

---

### W5. Проверка языка по доле кириллицы — наивная
```python
if ru < 30: raise ValueError(...)
```

30 символов кириллицы есть даже в посте, где 95% — английский. И для не-русских каналов проверка не работает вообще (CHANNEL_LANG != «русский»).

**Решение:** доля символов нужного алфавита > 60% от общего; для каждого языка — свой диапазон Unicode.

---

### W6. `parse_response` падает на markdown-разметке вокруг JSON
```python
if "```" in clean:
    for p in clean.split("```"):
```

Парсит ```json … ```, но если модель вернёт `text ```json{...}``` text` (что бывает редко, но бывает) — словит первый блок не туда.

**Решение:** регулярка `r"```(?:json)?\s*(\{.*?\})\s*```"` с `re.DOTALL`, fallback на `json.loads(raw)`.

---

### W7. `ensure_correct_link` удаляет ВСЕ `<a>` теги
Если модель в крючке вставит, например, `<a href="...">VOICE</a>` для красоты — её сотрут. Для текущих промптов это не проблема, но при расширении форматов (T4.1) — узкое место.

**Решение:** удалять только теги, чей `href` ≠ нужный, либо добавлять ссылку без удаления существующих.

---

### W8. `feedparser.parse` без таймаута и без User-Agent
Зависание на медленных RSS-серверах задержит весь job. Некоторые сайты блокируют пустой UA.

**Решение:** скачивать через `requests.get(timeout=15, headers={"User-Agent": "..."})` и передавать байты в `feedparser.parse(bytes_data)`.

---

### W9. Все статьи берутся `[:5]` подряд
```python
for entry in feed.entries[:5]:
```

Топ-5 свежих по каждому фиду — но если фид редко обновляется, мы каждый раз сканируем одни и те же 5. ОК для текущей логики (есть `posted_ids`), но в логах будет много шума и впустую сожжённые квоты GPT на фильтр.

**Решение:** сначала отфильтровать по `posted_ids`, потом брать первые N **новых** — экономия GPT-вызовов.

---

### W10. Workflow всегда `git push || true` — глотает ошибки push
Если push упал (например, force-push конфликт), workflow завершится зелёным, а данные не сохранятся. Через час всё повторится.

**Решение:** убрать `|| true` после стабилизации; логировать неуспешные push.

---

### W11. Нет `.gitignore`
`__pycache__`, `.env`, `venv/`, IDE-файлы — всё попадёт в репо при локальной разработке.

**Решение:** добавить стандартный Python `.gitignore`.

---

### W12. Нет README с инструкцией запуска
Файл `README.md` — пустой (2 байта). Любой новый разработчик / форк = тыкать в код.

**Решение:** базовый README со схемой работы, ссылками на CLAUDE.md / ROADMAP.md / SETI_INSTRUKCIYA.md.

## 🟢 ОПЦИОНАЛЬНЫЕ улучшения

- **O1.** `requirements.txt` без `--require-hashes` и lock-файла — версии плавают (`>=`). Перейти на `pip-tools` или `uv` с lock.
- **O2.** Магические числа разбросаны (`1024`, `4096`, `500`, `10`, `30`). Вынести в `config.py`.
- **O3.** Эталонные few-shot примеры — всегда 3 одних и тех же. Через несколько недель промпт «застынет» в одном стиле. Хранить пул из 10+ примеров, выбирать случайные 3.
- **O4.** Картинки 1080×1080 без кеша → каждый раз `pollinations.ai` генерирует заново. Можно сохранять в репо/CDN.
- **O5.** Нет метрик: сколько постов в день, % одобрений, среднее время до одобрения. Решается после T3.5.
- **O6.** Нет dry-run режима (сгенерировать пост, но не отправлять модератору). Полезно для отладки промптов.
- **O7.** В `requirements.txt` лежит `openai>=1.30.0`, но он не используется (всё через `requests`). Удалить.

## ✅ Что в коде сделано хорошо

- Чёткая структура: фильтр → определение рубрики → генерация → ensure_link → отправка.
- Few-shot примеры в промпте — рабочая практика, видно что промпт продуман.
- `ensure_correct_link` — отличная защита от типичной ошибки моделей.
- Двухступенчатый фильтр HIGH/MEDIUM/LOW — правильное решение для качества.
- Универсальность через env vars (`CHANNEL_TOPIC`, `CHANNEL_NICHE`) — готов к масштабированию.
- Fallback на текстовое сообщение, если `sendPhoto` упал — устойчивость.
- `concurrency` в workflow — попытка избежать гонок (хоть и неполная).
- Stash/pull/pop в workflow — единственно правильный способ работать с git в Actions.
